  • The robot is animated using CSS3 Animations as follows:
    • A stylesheet is appended to the document, containing@-webkit-keyframesdeclarations.
    • When the robot "program" runs, the keyframes in that stylesheet are programmatically modified, depending on the current action, to define the animations.
    • A-webkit-animation-nameproperty is temporarily applied to the robot sprite to run the animation.
    • Once the animation is complete (detected by a listener on thewebkitAnimationEndevent for the robot sprite), the new position and rotation are permanently applied to the sprite. This allows the keyframes declaration to be modified and applied again later.
  • The small "throb" on the buttons, the robot crash/explosion, and the "Waiting for program" message pulse are also done with CSS3 Animations.
  • As this demo relies on vendor prefixes (-webkit) to specify the animation rules, it won't work on non-WebKit browsers like Firefox. It could be modified to work on those browsers, though it would make the code more complex.

CSS3 transitions enable the properties of DOM elements to change gradually over time, rather than instantaneously as they do in CSS2.

This demo uses x* and y* classes on the tiles to set their positions in terms ofleftandrightCSS properties. All tiles also havetransition-*properties which specify how theleftandrightproperties change over time:

      /* property name and easing function */
      transition: left ease-out, top ease-out;
      -webkit-transition: left ease-out, top ease-out;

      /* how long a transition lasts when a property changes */
      transition-duration: 0.25s;
      -webkit-transition-duration: 0.25s;

To move a tile, the x* and y* classes of the empty tile and a clicked adjacent tile are swapped (via JavaScript). The animation is then done by the web runtime using hardware acceleration, if available.

In this demo, the transition durations can be uniform, as the distances travelled for a single move are constant. However, where more dynamic or complex animations are required, CSS3 animations may be more appropriate.

The banner (displayed when you complete the puzzle) is also animated via CSS transitions, triggered by adding anoverlayclass to a div element. To slide the banner out again, theoverlayclass is just removed.

Some of the UI elements need to be disabled while animations are occurring, to prevent the user clicking on a tile which is moving, or resetting the puzzle while it is shuffling. The demo adds a listener to thewebkitTransitionEndevent on the empty tile to manage button enabling/disabling, and to check for whether the puzzle is completed after the empty tile moves.

Note that the background image of each tile is separate from the position information, so that the correct part of the image is shown on each tile. A single image is used, but "sliced" so that the appropriate part is shown on each tile, using CSSbackground-positionproperties.
